# 寻根溯源 (RootsTrace) 项目第一阶段总结

## 📝 任务目标
完成「寻根溯源」族谱管理系统的基础框架搭建、数据库设计、大数据量性能模拟以及后端核心递归查询接口的打通。

## 🏗️ 已完成工作

### 1. 环境与基础设施搭建
- **WSL 环境集成**: 在 WSL (Ubuntu) 环境下成功安装并配置了 PostgreSQL 16 数据库。
- **后端框架**: 搭建了 Spring Boot 3.2 + MyBatis-Plus 3.5.7 的后端基础骨架，适配了 Java 17 和 Jakarta EE 标准。
- **前端框架**: 初始化了 React 18 + Vite + TypeScript 前端环境，集成了 Axios 和 Zustand。

### 2. 数据库设计与大数据量模拟
- **Schema 设计**: 实现了支持树形结构的 `members` 和 `relations` 表，包含外键约束、`TIMESTAMPTZ` 时区支持以及针对姓名搜索的 `pg_trgm` 索引。
- **大数据量填充**: 编写并成功运行了 Python 脚本，通过 `COPY` 指令批量导入了 **10 万级** 成员数据：
  - 其中 **Family 1 包含 55,000 名成员**，传承代数达 **32 代**，用于压力测试。

### 3. 后端核心逻辑实现
- **递归查询 (CTE)**: 在 `RelationMapper.xml` 中实现了基于 PostgreSQL 递归公用表表达式的 SQL 逻辑：
  - **祖先追溯**: 向上递归查询所有历代祖先。
  - **子孙查询**: 向下递归查询所有直系后代。
  - **亲缘路径**: 使用双向 BFS 算法查找两人间的最短路径。
- **通用响应封装**: 实现了 `Result<T>` 统一响应格式。

### 4. 关键问题解决记录 (Troubleshooting)
- **SB3 兼容性**: 修复了 `javax` 到 `jakarta` 包名变更导致的编译错误。
- **MyBatis-Plus 启动 Bug**: 通过升级到 `3.5.7` 解决了 Spring Boot 3.2 的 `factoryBeanObjectType` 类型异常。
- **XML 解析错误**: 解决了 Mapper 文件中 SQL 符号（`<`, `->`）导致的 XML 解析崩溃。
- **类型匹配**: 解决了 `TIMESTAMPTZ` 对应 `LocalDateTime` 的映射冲突，统一使用 `OffsetDateTime`。
- **双重初始化**: 关闭了 Spring Boot 的自动 SQL 初始化，避免与手动导入的大数据产生冲突。

## 🚦 当前状态
- **后端**: 运行在 `8080` 端口，核心数据接口 `api/members` 和 `api/query` 已跑通。
- **数据库**: PostgreSQL 稳定运行，已包含 10 万条生产级模拟数据。
- **前端**: 环境就绪，API 拦截器已配置，等待接入可视化组件。
